## Роль

Сервер **сам геокодит** — вся логика «координаты → каналы» живёт здесь (в
`ether-research` она прототипировалась на JS и переезжает сюда). Он принимает
координаты пользователя и вычисляет набор **ID каналов** (формат — контракт в
parent: ISO 3166-1/-2 для страны/области, `osm_type/osm_id` для
города/района/квартала). Сервер:

- геокодит координаты в набор ID каналов;
- хранит сообщения, ключ = ID канала (админ-ключ);
- держит WebSocket-соединения и рассылает сообщения подписчикам канала;
- управляет подпиской: при коннекте — на текущий набор каналов; при движении
  пользователя — переподписка только на изменившиеся уровни;
- отдаёт REST (`rest.go`) там, где не нужен пуш/сокет как объект: проверка и
  восстановление сессии, согласие с правилами, история сообщений.

Внутри хранения и рассылки канал — это просто **строка-ключ**, но саму эту строку
из координат вычисляет сервер.

## Контракт с клиентом

Клиент шлёт серверу свои координаты; сервер через геокодинг вычисляет набор ID
каналов для точки. Важное следствие однозначности из parent: два клиента в одном
месте дают одинаковые ID → попадают в один чат.

## Что предстоит

- [x] Выбор языка/стека (Go) и каркас сервиса
- [x] Геокодинг «координаты → набор ID каналов» (`nominatim.go`: reverse +
      details, слоты по `rank_address`)
- [x] Хранение сообщений по ID канала (SQLite в `store.go`: запись при
      `publish`, догрузка `GET /history` страницами по `before_id`)
- [x] WebSocket: подписка на набор каналов при коннекте
- [ ] Переподписка при смене единиц — диффом, только изменившиеся уровни
- [x] Идентификация: вход через Telegram (deep-link боту, `/start <токен>`,
      long-poll `getUpdates`) — единственное, что осталось асинхронным push на
      WS (`authed`); всё остальное про сессию — REST.
- [x] Персистентные сессии: пользователи и токены сессий в SQLite (`store.go`),
      восстановление через `POST /session/resume` после реконнекта (REST, не
      WS-кадр — не держит сокет ради разовой проверки). WS-подключение само
      аутентифицируется через `?token=` при апгрейде (см. `wsHandler` в
      `main.go`), без отдельного кадра входа. Логаута/TTL пока нет.
- [x] Согласие с правилами эфира — `users.rules_accepted_at` в SQLite, привязано
      к Telegram-аккаунту (`POST /rules/accept`, REST, требует токен в теле);
      отдаётся в `authed`/resume-ответе полем `rules_accepted`, чтобы клиент
      показывал экран правил не больше одного раза за жизнь аккаунта. Версии
      правил нет — приняв однажды, принял навсегда; понадобится `rules_version`,
      если текст правил когда-нибудь изменится.
- [x] REST-слой (`rest.go`) вынес resume/accept_rules/history из WS: это
      синхронный запрос-ответ без побочных эффектов на сокете — держать ради
      них открытый Client в хабе не нужно. `publish`/`locate`/`login_telegram`
      остались на WS осознанно (пуш и подписка сокета — см. PROTOCOL.md).
