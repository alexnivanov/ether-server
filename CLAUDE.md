## Роль

Сервер **сам геокодит** — вся логика «координаты → каналы» живёт здесь (в
`ether-research` она прототипировалась на JS и переезжает сюда). Он принимает
координаты пользователя и вычисляет набор **ID каналов** (формат — контракт в
parent: ISO 3166-1/-2 для страны/области, `osm_type/osm_id` для
города/района/квартала). Сервер:

- геокодит координаты в набор ID каналов;
- хранит сообщения, ключ = ID канала (админ-ключ);
- держит WebSocket-соединения и рассылает сообщения подписчикам канала;
- управляет подпиской: при коннекте — на текущий набор каналов; при движении
  пользователя — переподписка только на изменившиеся уровни;
- отдаёт REST (`rest.go`) там, где не нужен пуш/сокет как объект: проверка и
  восстановление сессии, согласие с правилами, история сообщений.

Внутри хранения и рассылки канал — это просто **строка-ключ**, но саму эту строку
из координат вычисляет сервер.

## Контракт с клиентом

Клиент шлёт серверу свои координаты; сервер через геокодинг вычисляет набор ID
каналов для точки. Важное следствие однозначности из parent: два клиента в одном
месте дают одинаковые ID → попадают в один чат.

## Как устроено (заметки)

- **Вход** — нативный Telegram Login SDK на клиенте отдаёт OIDC ID-token (JWT);
  сервер проверяет его подпись по публичным ключам Telegram (JWKS) и клеймы
  (`POST /auth/telegram`, `telegram.go`). Секретный токен бота серверу не нужен;
  сети к Telegram нет (JWKS тянется лениво). Гостевого режима нет — `publish`
  только на authed-сокете.
- **Сессии** — токен в SQLite (`store.go`), выдаётся в ответе `/auth/telegram`;
  повторный вход — `POST /session/resume`, логаут — `POST /session/logout`
  (идемпотентно, один токен). WS сам authed через `?token=` при апгрейде
  (`wsHandler`), отдельного кадра входа нет. TTL токенов пока нет.
- **Согласие с правилами** — `users.rules_accepted_at`, привязано к аккаунту
  (`POST /rules/accept`); в ответе про личность — поле `rules_accepted`. Версий
  правил нет — понадобится `rules_version`, если текст изменится.
- **Сообщения** — таблица `messages` хранит только `tg_id` автора (не имя);
  имя (`full_name`) и `avatar_url` для отдачи клиенту берутся `JOIN` из `users` (история) или
  из авторского соединения (live). Так фото/имя всегда актуальны, без дублей в
  каждой строке. Миграций схемы нет — до closed-beta MVP БД пересоздаём с нуля.
- **REST vs WS** — resume/logout/accept_rules/history и вход вынесены в REST
  (`rest.go`): это синхронный запрос-ответ без побочных эффектов на сокете. На WS
  осознанно остались только `locate` (подписка) и `publish`/`message` (рассылка).
